# 多账号与浏览器指纹隔离

本项目支持同时管理多个 V2EX 账号，并为每个账号提供**独立、确定性的浏览器指纹和行为参数**，
降低多账号在浏览器层被简单关联的概率。

> ⚠️ 多账号自动化属于较激进的玩法，V2EX 明确禁止刷活跃度。指纹隔离、错峰执行、行为参数差异
> 都只能降低部分关联特征，不能消除封号风险。请自行评估账号数量、频率和出口 IP，**后果自负**。

---

## 一、核心概念：Profile

通过环境变量 `V2EX_PROFILE` 指定当前操作的账号。profile 名仅允许字母、数字、下划线和连字符，也不能使用 `acc1` / `Acc1` 这类仅大小写不同的名称，或 `CON`、`NUL`、`COM1` 等 Windows 设备保留名。配置了 `V2EX_PROFILE_LIST` 后，签到和阅读子进程必须明确携带 profile，缺失时拒绝运行。

| 资源 | `default`（不设 profile） | `<profile>`（如 `acc1`） |
|------|--------------------------|--------------------------|
| Cookie 文件 | `~/.v2ex_cookie` | `~/.v2ex_cookie.<profile>` |
| Chrome 用户数据目录 | `reader/data/chrome-profile/default` | `reader/data/chrome-profile/<profile>` |
| 余额记录 | `balance_log.json` | `balance_log.<profile>.json` |
| 余额检查状态 | `balance_status.json` | `balance_status.<profile>.json` |
| 阅读队列 | `queue.db` | `queue.<profile>.db` |
| 账号身份哈希 | `profile_identity.json` | `profile_identity.<profile>.json` |
| 浏览器指纹 | 由 `default` 种子生成 | 由 `<profile>` 种子生成 |
| 行为参数 | 由 `default` 种子生成 | 由 `<profile>` 种子生成 |

> 单账号运行时显式设置 `COOKIE_FILE` 或 `DB_PATH` 会覆盖对应路径。Bot 多账号子任务和非默认 systemd 单元会使用自动派生路径，避免多个账号重新共享同一文件。

Telegram 授权文件、日志级别、综合阅读日志和全局阅读锁属于控制状态，所有 profile 共享；全局锁确保同一时间只启动一个阅读进程。升级前已有的共享余额和队列无法可靠判断属于哪个账号，因此保留给 `default`，不会自动复制或迁移给其他 profile。

`V2EX_COOKIE` 只允许初始化显式 `V2EX_PROFILE` 指定的当前账号，不会复制到列表其他账号；多账号模式没有指定 profile 时会忽略该启动凭证。其他账号应通过 Telegram 面板分别导入。

Cookie 文件分开只是第一层隔离。导入时还会先访问认证页确认真实账号，并将加盐身份哈希绑定到 profile。候选 Cookie 验证成功后整体替换，绝不会保留另一个账号缺失的认证字段；账号变化需要二次确认并清空该 profile 的旧 Chromium 目录。旧版本升级后首次建立身份绑定时也会重建该 profile 的 Chromium 状态，避免 localStorage、IndexedDB 等遗留凭证混入。身份验证失败时原 Cookie 保持不变。

## 二、指纹隔离做了什么

`reader/fingerprint.js` 以 **profile 名为种子**，确定性地生成一套指纹，保证：

- **同一依赖版本和配置下**，同一账号每次启动使用确定性参数；升级 Playwright/Chromium 后 UA 会随浏览器版本更新；
- **不同 profile 使用独立种子**生成 UA、视口、时区、语言、硬件参数和 WebGL 组合；候选池有限，因此不承诺任意两个 profile 的所有维度绝不重复。

隔离维度：

| 维度 | 说明 |
|------|------|
| User-Agent | 自动读取当前 Playwright 自带 Chromium 主版本 + Windows / macOS 平台 |
| 视口尺寸 | 常见桌面分辨率（1280×720 ~ 1920×1080） |
| 时区 | 美区常见时区：New_York、Los_Angeles、Chicago、Denver、Phoenix |
| 语言 | 以 en-US 为主（美区环境），部分含 zh-CN 辅助，含 `Accept-Language` |
| navigator.platform | Win32 / MacIntel |
| hardwareConcurrency | 4 / 8 / 12 / 16 |
| deviceMemory | 4 / 8 (GB) |
| WebGL vendor/renderer | Intel / NVIDIA / AMD / Apple 常见显卡 |
| navigator.webdriver | 隐藏（伪装为非自动化） |

候选值限制在常见桌面范围，但任何指纹改写都可能与字体、图形栈或出口地区产生交叉差异，不能视为低风险或反检测保证。

> 💡 **零额外开销**：指纹隔离仅在启动时做一次哈希计算（微秒级），不引入任何新依赖，
> 也**不会多开浏览器**。单账号或多账号串行运行时，资源占用与未启用指纹隔离时**完全相同**。

## 三、行为参数差异

v1.3.2 起，`reader/behavior.js` 还会基于 profile 名生成一组**确定性行为参数**：

- 单篇停留时长倍率；
- 偶尔长读概率的小幅偏移；
- 帖子间拟人间隔倍率；
- 余额检查间隔的小幅偏移。

这些参数只在常见范围内轻微变化，同一 profile 每次运行保持一致，不会每次启动随机漂移。
如果你显式设置了 `READ_DWELL_*`、`READ_HUMAN_GAP_*`、`READ_MEMORY_SETTLE_MS` 或
`BALANCE_CHECK_INTERVAL`，则以你的环境变量为准，不再由 profile 自动调整。

默认参数还会保证“最短拟人间隔 + 内存等待”不少于 8000ms；显式设置最短间隔或内存等待时允许覆盖这个默认下限。

> 这不是“反检测保证”。它只避免多个账号呈现完全相同的节奏参数；同 IP、多账号、过高频率、
> 长时间自动阅读仍然可能触发风控。

## 四、使用示例

### 1. 为每个账号保存 Cookie

```bash
cd checkin

# 账号 1（默认 profile）
V2EX_COOKIE="账号1的cookie" node v2ex-checkin.js --save-cookie
# → 保存到 ~/.v2ex_cookie

# 账号 2
V2EX_PROFILE=acc2 V2EX_COOKIE="账号2的cookie" node v2ex-checkin.js --save-cookie
# → 保存到 ~/.v2ex_cookie.acc2

# 账号 3
V2EX_PROFILE=acc3 V2EX_COOKIE="账号3的cookie" node v2ex-checkin.js --save-cookie
# → 保存到 ~/.v2ex_cookie.acc3
```

### 2. 多账号签到

```bash
cd checkin
node v2ex-checkin.js                 # 账号1
V2EX_PROFILE=acc2 node v2ex-checkin.js   # 账号2
V2EX_PROFILE=acc3 node v2ex-checkin.js   # 账号3
```

### 3. 多账号自动阅读（每个账号独立指纹与行为参数）

```bash
cd reader
node main.js                         # 账号1，指纹 A
V2EX_PROFILE=acc2 node main.js       # 账号2，指纹 B
V2EX_PROFILE=acc3 node main.js       # 账号3，指纹 C
```

> 启动时日志会打印当前 profile、指纹摘要与行为参数摘要，便于核对。

### 4. Telegram 多账号控制

配置 `V2EX_PROFILE_LIST=default,1,2` 后，每个 profile 就是一个固定账号位置。`/start` 会显示当前位置和脱敏用户名，面板中的余额、签到和阅读默认操作当前账号；通过“切换账号”可以更换当前位置。已有账号显示脱敏用户名和位置，缺少 Cookie 的位置显示“空”，点击后直接进入 Cookie 导入。阅读随后可选择 5、10、50 或 250 篇。文本命令也支持：

旧版本生成的身份文件没有脱敏显示提示时，面板会暂时显示“已登录”；下次 Cookie 导入、签到或阅读成功验证身份后会自动补全，不会为显示用户名额外发起网络请求。

```text
/sou acc2
/checkin acc2
/read acc2 50
/cookie acc2 A2=...
```

列表只有一个 profile 时不会增加多余选择步骤。面板中的空位置可以先选位置再粘贴 Cookie；直接粘贴 Cookie 且存在多个位置时，也可以再选择保存位置。每次导入使用独立随机事务号，候选内容只在 Bot 内存保留 5 分钟。旧消息、过期按钮、Bot 重启前按钮和列表重排后的选择都会失效。默认关闭的实验性飞书交互 Bot 仍只操作显式 `V2EX_PROFILE` 指定的单个账号。

内置调度器从本机时间 09:10 建立固定窗口；时段面板会逐段显示起止时间，已有账号的位置标为“有任务”，缺少 Cookie 的位置标为“空”。前一个账号提前完成后会等待下一边界，签到耗时会从阅读预算扣除。`profile 数量 × PROFILE_TIME_SLOT_HOURS` 不得超过 24；`/stop [profile]` 会停止当前子进程并取消后续串行账号。

## 五、重要建议

- **强烈建议错开时间**运行各账号，不要并发同时跑，行为更自然，也省内存（每个阅读实例都会起一个 Chromium）。
- **建议每个账号配独立出口 IP**（不同代理/VPS）。指纹隔离能降低浏览器层关联，
  但同 IP 多账号仍是最容易被关联的特征，指纹隔离无法解决 IP 维度的问题。
- 自动阅读本身属于高风险行为。账号越多、阅读越密、出口越集中，封号风险越高。请克制。

### 多账号串行运行示例脚本

```bash
#!/usr/bin/env bash
# run-all.sh —— 依次为多个账号签到（错峰可加 sleep）
set -e
for p in default acc2 acc3; do
  echo "===== profile: $p ====="
  V2EX_PROFILE="$p" node checkin/v2ex-checkin.js || true
  sleep $((RANDOM % 120 + 30))   # 随机间隔 30~150 秒，行为更自然
done
```

## 六、配合定时任务

### systemd timer（每个账号独立单元）

用一键脚本为每个 profile 生成独立的 timer，并用不同时间错峰：

```bash
sudo bash scripts/install-systemd.sh                  # 账号1（default）
sudo bash scripts/install-systemd.sh --profile acc2   # 账号2
sudo bash scripts/install-systemd.sh --profile acc3   # 账号3
```

> 脚本运行时会**交互式询问**各任务的执行时间。阅读任务共享一个全局锁，systemd 单元不会自动排队；账号间必须按可能的最长阅读窗口错开（默认可按约 4 小时），仅错开几分钟会使后启动的阅读任务因忙碌退出。需要固定时段自动串行时，使用 Docker/Render 的 Bot 内置调度器和 `V2EX_PROFILE_LIST`。

每个 profile 会生成 `v2ex-checkin-<profile>`、`v2ex-ping-<profile>`、`v2ex-reader-<profile>` 等独立状态单元；`RandomizedDelaySec` 会在设定时间上再叠加随机抖动，因此预留窗口时也要包含该抖动。查看：

```bash
systemctl list-timers 'v2ex-*'
```

### 回退：crontab（无 systemd 时）

```cron
# 按服务器本机时间错峰：账号1 09:10、账号2 09:25、账号3 09:40
10 9 * * * /usr/bin/node /path/to/checkin/v2ex-checkin.js >> /var/log/v2ex.log 2>&1
25 9 * * * V2EX_PROFILE=acc2 /usr/bin/node /path/to/checkin/v2ex-checkin.js >> /var/log/v2ex.log 2>&1
40 9 * * * V2EX_PROFILE=acc3 /usr/bin/node /path/to/checkin/v2ex-checkin.js >> /var/log/v2ex.log 2>&1
```
