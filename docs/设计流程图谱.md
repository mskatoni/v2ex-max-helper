# V2EX Max Helper 设计流程图谱

本文描述 `v1.4.11` 的目标设计，用来回答“系统应该怎样运行”。真实代码路径和差异见 [实际流程图谱.md](实际流程图谱.md)。

## 设计原则

- 所有入口共用一套配置解析和 profile 命名规则。
- 最多管理 6 个 profile；账号状态隔离，控制状态共享。
- 同一时间最多运行一个阅读进程，多账号按时间窗口串行。
- 调度时间按运行环境的本机时区计算。
- 代理默认关闭；显式开启后配置错误必须拒绝启动，不能回退直连。
- Telegram 是主要控制面；飞书默认关闭，仅保留实验入口。

## 总体架构

```mermaid
flowchart TD
    D0["Docker、Render 或 VPS"] --> C0["统一加载环境变量和配置文件"]
    C0 --> P0{"是否启用代理"}
    P0 -->|"否"| N0["使用 HTTPS 直连"]
    P0 -->|"是且配置合法"| N1["使用本机或获准的 LAN 代理"]
    P0 -->|"是但配置非法"| X0["拒绝启动"]
    N0 --> B0["启动 Telegram Bot 和调度器"]
    N1 --> B0
    B0 --> S0["自动调度入口"]
    B0 --> M0["手动控制入口"]
    S0 --> L0["全局任务互斥"]
    M0 --> L0
    L0 --> A0["选择并加载 profile"]
    A0 --> K0["签到和保活"]
    A0 --> R0["自动阅读"]
    K0 --> ST0["写回 profile Cookie"]
    R0 --> ST0
    R0 --> DB0["写入 profile 队列和余额状态"]
    K0 --> Q0["通知层"]
    R0 --> Q0
```

## 自动调度设计

```mermaid
flowchart TD
    T0["每分钟读取本机时间"] --> M0{"是否配置 profile 列表"}
    M0 -->|"否"| S1["09:10 运行单账号签到"]
    S1 --> S2["09:15 运行单账号阅读"]
    M0 -->|"是"| W0["09:10 进入多账号时间窗口队列"]
    W0 --> W1["等待 profile 1 窗口开始"]
    W1 --> C1["profile 1 签到"]
    C1 --> R1["profile 1 阅读"]
    R1 --> W2["等待 profile 2 窗口开始"]
    W2 --> C2["profile 2 签到"]
    C2 --> R2["profile 2 阅读"]
    R2 --> WN["依次处理后续 profile，最多 6 个"]
    T0 --> P0{"本机时间是否为 00:00、06:00、12:00 或 18:00"}
    P0 -->|"是"| PK["按 profile 串行保活"]
    P0 -->|"否"| T0
```

设计中的“时间窗口”表示每个 profile 有明确的计划开始边界和最长运行时长。前一个账号提前结束时，下一个账号仍应等到自己的窗口开始，避免所有账号在短时间内连续启动。

## Telegram 控制设计

```mermaid
flowchart TD
    U0["授权用户发送命令或点击按钮"] --> A0{"操作类型"}
    A0 -->|"签到"| P0["选择 profile"]
    A0 -->|"阅读"| P1["选择 profile"]
    A0 -->|"余额"| P2["选择 profile"]
    A0 -->|"Cookie"| P3["选择 profile 或先解析 Cookie"]
    P0 --> C0["检查 Cookie 和任务互斥"]
    C0 --> C1["启动该 profile 签到"]
    P1 --> R0["选择 5、10、50 或 250 篇"]
    R0 --> R1["检查 Cookie 和全局阅读锁"]
    R1 --> R2["启动该 profile 阅读"]
    P2 --> B0["读取该 profile 本地余额记录"]
    P3 --> K0["按随机事务号暂存候选 Cookie，最长 5 分钟"]
    K0 --> K1["访问认证页确认真实账号"]
    K1 --> K2{"是否与 profile 已绑定身份一致"}
    K2 -->|"一致或首次绑定"| K3["原子替换 Cookie 和身份哈希"]
    K2 -->|"不同"| K4["二次确认换绑并清理旧 Chromium 状态"]
    K4 --> K3
```

只有一个可用 profile 时，应直接进入操作，不增加 profile 选择步骤。回调数据携带短期会话 nonce 和 profile 下标，不携带 profile 名或 Cookie；旧会话、列表重排和 Bot 重启后的按钮必须失效。

## 阅读任务设计

```mermaid
flowchart TD
    R0["取得全局阅读锁"] --> Q0["加载 profile 队列"]
    Q0 --> B0["启动 Chromium 和 profile 指纹"]
    B0 --> C0["注入 Cookie 并建立余额基线"]
    C0 --> U0["获取一篇帖子"]
    U0 --> V0["页面停留和随机滚动"]
    V0 --> K0["同步 Cookie"]
    K0 --> M0["等待内存回收"]
    M0 --> H0["等待拟人间隔"]
    H0 --> E0{"本篇是否成功"}
    E0 -->|"是"| I0["队列计数加一"]
    E0 -->|"否"| F0["跳过异常 URL 并检查登录态"]
    I0 --> S0{"是否达到停止条件"}
    F0 --> S0
    S0 -->|"否"| U0
    S0 -->|"是"| Z0["最终余额检查"]
    Z0 --> Z1["关闭浏览器和队列"]
    Z1 --> Z2["释放全局锁并发送结果通知"]
```

默认参数下，“最短拟人间隔 + 内存等待”不得低于 8000ms。用户显式设置最短间隔或内存等待时，视为高级覆盖。

## 状态边界

| 类型 | 按 profile 隔离 | 全局共享 |
|---|---|---|
| 登录状态 | Cookie、加盐账号身份哈希、Chromium 用户目录、凭证锁 | 无 |
| 阅读状态 | 队列数据库、余额记录、余额检查状态 | 阅读锁 |
| 行为身份 | 指纹、行为参数 | 无 |
| 控制状态 | 无 | Telegram 授权、日志级别、综合日志 |
| 调度状态 | profile 执行结果 | 固定窗口周期、取消状态 |
| 通知 | 消息中标明 profile | Telegram Chat ID、飞书配置 |

`default` 应继续使用旧文件名；非默认 profile 使用带名称的文件。无法确认归属的旧共享余额和队列保留给 `default`，不自动复制。

## 通知设计

通知层应覆盖 Cookie 失效、签到失败、余额变化、阅读异常和正常阅读完成。网络错误、超时或非 2xx 响应只记录脱敏状态，不得阻塞签到或阅读主流程。
